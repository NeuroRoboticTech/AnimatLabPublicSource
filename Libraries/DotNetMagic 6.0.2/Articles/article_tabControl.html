TabControl Overview
The TabControl is used to manage a collection of TabPage instances which themselves are used to describe an individual page. Only a single page can be displayed at any time and a set of tab headers are shown to allow the user to select between the available pages.

The following sections show how to use the control and describe the more important properties and events for controlling the TabControl. This is not however an exhaustive list and so the help document should be consulted for the definitive list of all functionality.

Creating your TabControl

Creating your TabControl from the Toolbox

You can use the designer to modify a TabControl instance along with its TabPage instances. If not already present then you will need to add the control to the toolbox by right-clicking the toolbox window and selecting the Add/Remove items... option. Navigate to the installation directory called Bin and select the DotNetMagic2005.dll library. The toolbox will now list the extra controls that are exposed by the library including the TabControl. Drag and drop a new instance onto your Form and select the TabPages property.

This will cause a dialog to appear that allows the creation and removal of pages to the control instance. It also allows you to modify the properties for each individual page. Finish defining the required pages and then exit the dialog. Notice that you can now operate the control in design mode by clicking on the tab headers to change page.

As the TabPage class is derived from the Panel class you can now drag and drop other controls onto the individual tab pages as desired. The designer will automatically generate the code needed to cause this configuration to be persisted as code in your Form.

Creating your TabControl manually

If you need to create your TabControl manually then use the following simple example: -

C#


   Crownwood.DotNetMagic.Controls.TabControl tab = 
      new Crownwood.DotNetMagic.Controls.TabControl();
VB.NET

   Dim tab As New Crownwood.DotNetMagic.Controls.TabControl()
NOTE:

Because the control class is named TabControl and this is the same name as an existing control in the System.Windows.Forms namespace, you will need to fully qualify the name of the class to be created. Otherwise the compiler will not be able to choose between the two available classes with the same names. This is also true of the TabPage class and hence the use of fully qualified names in all the example code.

TabPages collection
The TabControl exposes a property called TabPages that allows the developer to add, remove and modify pages displayed by the control. Use the standard collection methods and properties to interact with the collection at runtime.

Here is an example that shows the collection being manipulated: -

C#


   // Create a new TabControl instance
   Crownwood.DotNetMagic.Controls.TabControl tab = 
      new Crownwood.DotNetMagic.Controls.TabControl();

   // Create two pages for the TabControl
   Crownwood.DotNetMagic.Controls.TabPage tabPage1 = 
      new Crownwood.DotNetMagic.Controls.TabPage();
   Crownwood.DotNetMagic.Controls.TabPage tabPage2 = 
      new Crownwood.DotNetMagic.Controls.TabPage();

   // Add the new pages to the collection
   tab.TabPages.AddRange(
      new Crownwood.DotNetMagic.Controls.TabPage[]{tabPage1, tabPage2});

   // Remove all the contents
   tab.TabPages.Clear();

   // Add a single entry
   tab.TabPages.Add(tabPage1);

   // Insert a single entry at a given position
   tab.TabPages.Insert(0, tabPage2);

   // Use index accessor for retrieving a page reference
   Console.WriteLine("TabPages[0] name = {0}", tab.TabPages[0].Text);

   // Test if the collection contains an entry
   Console.WriteLine("Contains tabPage1 = {0}", tab.TabPages.Contains(tabPage1));

   // Gain access to a Page by its Title
   Console.WriteLine("Contains a named page? {0}", tab.TabPages["Page1"] != null);

   // Remove by object reference
   tab.TabPages.Remove(tabPage1);

   // Remove by index
   tab.TabPages.RemoveAt(0);
VB.NET

   ' Create a new TabControl instance
   Dim tab As New Crownwood.DotNetMagic.Controls.TabControl()

   ' Create two pages for the TabControl
   Dim tabPage1 As New Crownwood.DotNetMagic.Controls.TabPage()
   Dim tabPage2 As New Crownwood.DotNetMagic.Controls.TabPage()

   ' Add the new pages to the collection
   tab.TabPages.AddRange( _
      new Crownwood.DotNetMagic.Controls.TabPage(){tabPage1, tabPage2})

   ' Remove all the contents
   tab.TabPages.Clear()

   ' Add a single entry
   tab.TabPages.Add(tabPage1)

   ' Insert a single entry at a given position
   tab.TabPages.Insert(0, tabPage2)

   ' Use index accessor for retrieving a page reference
   Console.WriteLine("TabPages[0] name = {0}", tab.TabPages(0).Text)

   ' Test if the collection contains an entry
   Console.WriteLine("Contains tabPage1 = {0}", tab.TabPages.Contains(tabPage1))

   ' Gain access to a Page by its Title
   Console.WriteLine("Contains a named page? {0}", _
      Not tab.TabPages("Page1") Is Nothing)

   ' Remove by object reference
   tab.TabPages.Remove(tabPage1)

   ' Remove by index
   tab.TabPages.RemoveAt(0)
TabPage Class
The TabPage class derives from the Panel base class and so can be used in the designer as a surface for placing controls onto. This allows the developer to drag and drop controls from the toolbox onto the individual pages of the TabControl to achieve the required runtime layout.

However, you are not obliged to use the TabPage object itself as the control for display when the page is selected. Instead you may assign any Control derived instance to the TabPage.Control property which will then be displayed whenever the tab is selected. This is particularly useful when you do not know the control that will be displayed until runtime, or if you need to show a control that derives from UserControl or Form.

This property has the added benefit of allowing the creation of more flexible designs that achieve greater code reuse. Placing controls onto a TabPage surface is restrictive as it implies the layout can only be used within a TabControl. Instead create a Form derived class that contains the layout of controls as desired along with any associated control logic. When this layout is needed inside a TabControl simply create a TabPage instance along with an instance of your new class which is then assigned to the TabPage.Control property.

You now have the added benefit that your new class can be reused. Anywhere you need the same user interface layout you just instantiate a new instance and add as a child form. Alternatively you might derive a class that adds 'Ok' and 'Cancel' buttons and a few extra lines of code to create a dialog box.

Maybe the greatest benefit is the ability to change the TabPage.Control on the fly which will immediately change the control displayed in the TabControl. This gives maximum flexibility to the developer.

Text and Image

As you would expect for the design of any TabPage class you can define the text that is displayed for a particular instance. Assign a string to the TabPage.Title property.

An image can be displayed along with the title text by supplying appropriate values to the TabPage.ImageList and TabPage.ImageIndex properties. Note that the TabControl also has a property called ImageList that is used whenever a TabPage instance specifies an ImageIndex value but does not provide an ImageList reference value. If you know all your images come from the same strip then simply set the strip into the TabControl and then specify the correct image index into each TabPage instance. If the images come from several different strips then you can specify the strip and index with each individual page.

You are not restricted to just images, alternatively you can provide an Icon that always takes priority over any ImageList values that may have been specified.

Additional properties

If you need to assign a unique name per page then you can make use of the TabPage.UniqueName string property. This is useful if you have several pages with the same title text or where the text is localized for different languages. Using this property allows you to uniquely identify each page instance, but you need to assign the values yourself as the property defaults to false.

The developer can store page specific information relevant to their application using the TabPage.Tag property. No use is ever made of this by the TabControl or the TabPage classes. In the sample code example an integer value of 99 is boxed and stored for later use.

The TabPage.Selected property can be set to indicate if the page should be selected when it is added into the TabControl. Once added the value will reflect the current selected status of the page and can be set to make a page become the currently selected one. Note that you cannot set this value to false as the TabControl would not know which other page should become the newly selected one.

If your TabPage is quite complex it may have several child controls. In this case you might want to define which of the child controls starts with the focus when the page is selected. Use the TabPage.StartFocus property by assigning a reference to a child control. Note that when the user moves away from the current page it will update this property with the control that currently has focus so that when the user returns the focus will be restored to the same place it left off.

Appearance
The best way to become familiar with the different properties that control the appearance of the TabControl is via the SampleTabControl application that is supplied as part of the installation. This allows you to interact with each of the major settings and instantly see the visual changes it causes.

Appearance property

This is always the first property you should change because it automatically alters the value of other properties to the defaults for the selected appearance. The three possible values for the property are: -
  • MultiDocument
  • MultiForm
  • MultiBox
The MultiDocument appearance will place the page tabs at the top of the control and show both arrow and close buttons. An example use of this would be to control which of a group of documents is displayed. For example, use this appearance to achieve the same effect as the Visual Studio .NET control that contains the open documents.

The second appearance VisualAppearance.MultiForm places the page tabs at the bottom of the control and does not show any buttons. Instead all the tabs are sized to fit within whatever width the control is defined as. Use this appearance to mimic the style of tab control seen in Visual Studio .NET inside docking windows.

Use the third appearance VisualAppearance.MultiBox to create the high contrast appearance you get when editing an HTML document in Visual Studio .NET, where the document has two options at the bottom of the window.

Once the appearance has been set the developer can still modify the other properties to alter the exact look and feel wanted.

Style property

Second in importance is the Style property that is used to determine the look and feel of the control. This property is present on most of the DotNetMagic controls and allows a consistent look and feel to be achieved across your whole application. The six possible values for the property are: -
  • VisualStyle.Plain
  • VisualStyle.IDE2005
  • VisualStyle.Office2003
  • VisualStyle.Office2007Blue
  • VisualStyle.Office2007Silver
  • VisualStyle.Office2007Black
VisualStyle.Plain will not draw a border around the button until the mouse moves over the button area and then a border is drawn.

VisualStyle.IDE2005 will draw with the look and feel of the VS.NET 2005 environment.

VisualStyle.Office2003 will draw the button with the same style of gradient effect and colors as the standard Microsoft Office 2003 applications.

VisualStyle.Office2007Blue will draw with the Blue color scheme from Office 2007. VisualStyle.Office2007Silver will draw with the Silver color scheme from Office 2007. VisualStyle.Office2007Black will draw with the Black color scheme from Office 2007.

OfficeStyle

When using the one of the office styles this property is used to determine the colors used to draw the tab headers area. If you decide to override the individual colors then you can still do this by providing the appropriate value to the other color properties. But when the color properties are defaulted this style property is used to determine the color scheme.

ExtraOfficeTabInset

This property is only used when one of the office styles is defined for the Style property. It determines how many pixels inwards to place the first tab header. By default this will be zero and so the first tab header starts hard against the left edge of the control but you can specify a positive value to move it across.

OfficePixelBorder

This property is only used when one of the office styles is defined for the Style property. It will cause a single pixel border to be placed around the page area of the control and will join up with the separator line in the header.

OfficeHeaderBorder

This property is only used when one of the office styles is defined for the Style property. It will ensure that the pixel border is extended around the header area as well as the main page area when it is used in conjunction with the OfficePixelBorder property.

IDE2005Style

When using the VisualStyle.IDE2005 setting this property is used to determine the colors used to draw the tab headers area. If you decide to override the individual colors then you can still do this by providing the appropriate value to the other color properties. But when the color properties are defaulted this style property is used to determine the color scheme.

IDE2005ExtraTabInset

This property is only used when the VisualStyle.IDE2005 setting is defined for the Style property. It determines how many pixels inwards to place the first tab header. By default this will be zero and so the first tab header starts hard against the left edge of the control but you can specify a positive value to move it across.

IDE2005PixelBorder

This property is only used when the VisualStyle.IDE2005 setting is defined for the Style property. It will cause a single pixel border to be placed around the page area of the control and will join up with the separator line in the header.

IDE2005HeaderBorder

This property is only used when the VisualStyle.IDE2005 setting is defined for the Style property. It will ensure that the pixel border is extended around the header area as well as the main page area when it is used in conjunction with the IDE2005PixelBorder property.

IDE2005TabJoin

This property is only used when the VisualStyle.IDE2005 setting is defined for the Style property. When defined as False the selected tab header still has a separating line between itself and the main tab area.

PositionTop property

Use this property to determine if the tab headers are shown at the top or the bottom of the control instance. This value has a default value for each of the different Appearance property settings and so will be reset to the default whenever the Appearance property is changed. You can however change the value afterwards.

MultiLine property

When there are too many tabs to fit onto a single tab header line then this property will determine what action to take. If set then multiple lines are created and so the tab header area gets bigger and the space for tab page contents becomes smaller. You can use the associated

MultiLineFullWidth

property if you need to ensure that all the lines of tabs stretch across the full width of the tab header area.

ShrinkPagesToFit property

This property is used to determine if all the tab page headers should be squeezed into the available tab header width. In this case it will truncate the drawn text or even exclude drawing the text entirely to ensure that all the tab headers fit. When not defined then the arrow buttons can be used to scroll left and right to show those tab headers not visible.

ImageSize

The default size of images drawn in the tab headers is 16 x 16. You can use this property to change the size of the drawn images to whatever you require. Note that all tab headers will use the same size and you cannot set the size on a per page basis.

ShowClose and ShowArrows properties

You can override the decision to show the close and navigation arrow buttons by using these two properties. The properties values are set whenever the Appearance is defined but you can alter the values afterward to achieve a custom appearance and operation.
Colors

BackColor property

Use this property to determine the background drawing color for the control. By default this is defined to be SystemColors.Control but can be changed to achieve a custom look instead.

TextColor property

Color for drawing the title text of the currently selected page.

TextInactiveColor property

Color for drawing the title text of pages that are not selected.

HotTextColor property

Color used to highlight title text when the page is under the mouse and the HotTrack property set.

ButtonActiveColor property

Color used for drawing arrows/close buttons when they are active.

ButtonInactiveColor property

Color used for drawing arrows/close buttons when they are inactive.
Operation

HideTabsMode property


Determines when the tab headers area should be made visible. This can be very useful when you need to save as much screen real estate as possible. The four possible values for the property are: -
  • HideTabsModes.ShowAlways
  • HideTabsModes.HideAlways
  • HideTabsModes.HideUsingLogic
  • HideTabsModes.HideWithoutMouse
The HideTabsModes.ShowAlways setting is the default and will show the tab headers area all the time.

Tab headers are always hidden when the HideTabsModes.HideAlways mode is selected and is useful for creating workflow scenarios. Here you could create several pages and only switch them with programmatic control and therefore restrict the users workflow.

Alternatively you might only want the tab headers showing when there are two or more pages present in the control. If there are no pages or just a single page then you might not need the headers as there are no alternative choices of page for the user. In this case use the HideTabsModes.HideUsingLogic mode.

Another way to save space is the HideTabsModes.HideWithoutMouse setting. Here the tab headers are only drawn when the mouse moves over the area that would be occupied by the headers. So when the mouse is away from the control the headers are hidden to give as much space as possible to the selected page. In this mode use the LeaveTimeout property to define the delay between the mouse leaving the tab headers area and the tab headers being hidden again.

WithoutMouseVector property

When the HideTabsMode property is defined with HideWithoutMouse then the control will only show the tab headers when the mouse is over the control. Sometimes you might want to restrict the area of the control that allows the tab headers to be shown. Set this new property to the pixel length form the tab headers edge that should be the hot spot.

HoverSelect property

As the mouse moves over a tab header it will automatically become the selected page. You need to be careful when using this setting because if your setup changes the position of the tab header depending on which one is selected, then it can cause fast switching that is difficult for the user to operate.

ContextMenu property

Create and assign a ContextMenuStrip instance to the control and it will be shown whenever the user right clicks on a tab header. This ensures your context menu appears in the same style as the TabControl itself.

ToolTips property

If you hover the mouse over a tab header when this property is defined then a tool text will appear that is specific to the tab. You can define this text by assigning a string to the TabPage.ToolTip property.

TextTips property

If you hover the mouse over a tab header when this property is defined and the text for the header is not fully displayed then a tool tip will appear with the full text shown. This is useful when the user has reduced the width of display so that ellipses are shown for the tab header because of a shortage of space. This allows the user to hover the mouse over the tab and confirm the full name of the tab page. This property takes lower priority to the ToolTips property and so only displays when ToolTips are set to False.

AllowDragReorder

By default this property is defined as True and allows the user to drag an individual tab page header and move it to another position. This allows the user to reorder the pages as desired. Please note that at the moment this is only valid for single line controls and multiline controls will not allow reordering.

DragOutside

If you need to allow the user to drag a page out of the control then set this property to True. When the page is dragged out of the control it will disappear from display and PageDragStart and PageDragMove events be generated. When dragging finishes the PageDragEnd or PageDragQuit event are generated as appropriate.
TabStrip Simulation
The TabControl cannot be directly used as just a tab strip so that only the tab headers are displayed. But you can achieve the desired functionality via a different mechanism.

If you assign the same control to each of the TabPage instances then each page will display the same control when selected. This achieves the same effect as that desired from a tab strip, where you save on resources by showing the same control for each page but populate it with different contents depending on the selected page.

In fact you are not restricted to assigning the same control to each page. You can of course assign a different control to each page, the same control to all pages or any combination between. This gives as much flexibility as you could need from a separate tab strip style control.

Notifications
There are several interesting events that can be generated by the TabControl which allow its operation to be monitored and modified.

ClosePressed event

As you would expect, this event is fired when the user presses the close button. It is important to remember that the control will not remove the actual page itself. Therefore you must intercept this event and remove the selected page from the TabPages collection manually. This allows greater control over the operation as you can perform validation or other processing.

SelectionChanging/SelectionChanged events

When a change in selection is about to occur the SelectionChanging event is generated. This allows the client a chance to peek at the proposed change and perform additional processing. At this point the SelectedIndex and SelectedTab properties will still indicate the old page as being selected.

Once the change has actually occured the SelectionChanged event is fired and now the SelectedIndex and SelectedTab properties indicate the newly selected page.

PageGotFocus/PageLostFocus events

A useful feature of the control is the ability to generate events when the focus arrives and leaves the currently selected page. This allows processing to occur when the users attention shifts to your tab page which would prove otherwise to be far to expensive to conduct all the time.

NOTE:You should however note a word of caution on its use. The focus is monitored by hooking into the focus events of all controls on the tab page which works fine unless you are adding or removing controls to the page dynamically.

DoubleClickTab event

Whenever the user performs a mouse double click on a tab header it wil generate this event. This allows you to perform extra work over and above the standard change in selection that will occur automatically.

TooltipDisplay event

When a tooltip is about to be displayed this event is generated so that you can override the text that will be shown. You can also cancel the event and so prevent the tooltip from being displayed at all. This is particularly useful if you need to calculate the tooltip text on the fly.
